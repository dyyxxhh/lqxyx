## 2026-06-06 T7 Phaser 3 mobile/desktop input documentation research
- Official docs target: Phaser 3.90.0 API pages are versioned but marked no longer actively maintained; concepts pages are current 2026 docs and still document Phaser 3-style APIs. Primary docs: Input concept `https://docs.phaser.io/phaser/concepts/input`, Scale Manager concept `https://docs.phaser.io/phaser/concepts/scale-manager`, v3.90 InputPlugin `https://docs.phaser.io/api-documentation/3.90.0/class/input-inputplugin`, v3.90 ScaleManager `https://docs.phaser.io/api-documentation/3.90.0/class/scale-scalemanager`, v3.90 Pointer `https://docs.phaser.io/api-documentation/3.90.0/class/input-pointer`, KeyboardPlugin `https://docs.phaser.io/api-documentation/class/input-keyboard-keyboardplugin`.
- Desktop keyboard: use `scene.input.keyboard.createCursorKeys()` for arrow + space/shift polling and/or `scene.input.keyboard.addKeys('W,A,S,D,F,Q', true, false)` / keyed object form for WASD + interactions; poll `key.isDown` in `update()` for movement and use edge-triggered `keydown-F` / `keydown-Q` or just-down logic for interactions. Use `addCapture('UP,DOWN,LEFT,RIGHT,SPACE,W,A,S,D,F,Q')` or `enableCapture=true` to prevent browser scroll/defaults; note captures are global. Test implication: verify held-key diagonal movement and that input-lock state suppresses consumed movement/action without relying on DOM text.
- Pointer/touch joystick: Phaser unifies mouse and touch as pointer events (`pointerdown`, `pointermove`, `pointerup`, `pointerupoutside`) and sends `Pointer` objects with `x/y`, `worldX/worldY`, `downX/downY`, `getDistance()`, `getAngle()`, `isDown`, `id`, and DOM touch `identifier`. Implement joystick by owning the left-side touch pointer id/identifier on `pointerdown`, computing vector from fixed base to current pointer each move, clamping radius, quantizing to 8 directions, and clearing on matching `pointerup` or `pointerupoutside`.
- Multi-touch / `activePointers`: configure at game boot with `input: { activePointers: 2 }` or add at runtime with `scene.input.addPointer(n)`; docs note `scene.input.manager.pointersTotal` and `scene.input.pointer1..pointer10`, with additional pointers undefined until added. T7 needs at least two touch pointers for simultaneous left joystick + right interact button; tests should dispatch separate touch identifiers and assert debug state records both joystick pointer and interact pointer.
- Do not depend on `scene.input.activePointer` for the joystick while another finger taps interact: official docs define it as the current/most-recent active pointer. Track the joystick pointer by `pointer.id` / `pointer.identifier` and ignore other pointers for movement until cleanup.
- Pointer cleanup caveat: listen to both `pointerup` and `pointerupoutside`; optionally also clear on input `gameout`, scene shutdown, visibility loss, orientation changes, or when an input lock starts. This avoids stuck movement if a finger leaves the canvas, fullscreen changes, or tests synthesize incomplete touch sequences.
- Fullscreen: request with `scene.scale.startFullscreen()` / `toggleFullscreen()` only inside a real user gesture, preferably a `pointerup` handler on an explicit prompt button. Browser fullscreen cannot be forced; iframe fullscreen also needs `allowfullscreen`. Listen for `enterfullscreen`, `fullscreenfailed`, `fullscreenunsupported`, and `leavefullscreen`; store status in debug/UI state and let the player continue in non-fullscreen mode if refused.
- Orientation prompt: use `scene.scale.isPortrait`, `scene.scale.isLandscape`, and `scene.scale.on('orientationchange', ...)` to show a rotate-device overlay when portrait. `scene.scale.lockOrientation('landscape')` exists but should be treated as best-effort and not required for playability; tests should assert portrait prompt visibility/state separately from fullscreen success.
- Scale behavior for fixed 1280x720: keep `scale: { mode: Phaser.Scale.FIT, autoCenter: Phaser.Scale.CENTER_BOTH, width: 1280, height: 720 }`. Docs say FIT preserves aspect ratio by changing CSS display size while leaving game/base canvas dimensions fixed; `CENTER_BOTH` depends on parent sizing. Keep CSS parent viewport constraints from Task 6 and expose/verify `gameSize`, `baseSize`, `displaySize`, and canvas/client bounds in debug state for mobile tests.
- Scale/input caveat: Phaser docs call out `scene.scale.updateBounds()` when the canvas position/visibility changes outside Phaser because canvas DOM bounds are used for input coordinate transforms. T7 should call/update bounds after fullscreen/orientation/layout prompt transitions if pointer positions become offset.
- Input locks: Phaser provides broad switches (`scene.input.enabled = false`, `scene.input.keyboard.enabled = false`, individual `key.enabled = false`, `gameObject.disableInteractive()`), but for T7 prefer a central game input-lock predicate so UI prompts can still receive pointer events while movement/interact actions are ignored. Required lock reasons: dialogue, black screens, elevator fades, scripted movement, endings; lock start should zero joystick vector and clear pending one-shot interact.
- Mobile interaction design: use one on-screen interact button, not separate F and Q buttons. Its tap should resolve the current context in priority order from game state (e.g. current `Q` contextual branch if active, otherwise `F` proximity/action, or a scene-defined interaction resolver), so mobile remains equivalent without clutter. Tests should set/inspect current context and verify the single button dispatches the resolved action.
- Official docs were sufficient for core APIs; GitHub example searches for fullscreen/joystick patterns in Phaser examples were not necessary/reliable enough to supersede official docs.


## 2026-06-06  Playwright T7/T8/T15/T17 mobile + narrative UI research
- Classification: TYPE A conceptual research; official docs checked through Playwright 2026 docs sitemap and targeted pages. No production code/tests were inspected or changed; only this notepad was appended.
- Mobile landscape emulation: configure a Playwright project with `import { defineConfig, devices } from '@playwright/test'`, then spread a device descriptor such as `...devices['Pixel 5']` or current project choice and override `viewport` **after** the spread for landscape, e.g. `viewport: { width: 915, height: 412 }`; Playwright docs explicitly warn that device descriptors already define `viewport`, `isMobile`, and touch-related options, so overrides must come after the spread. Sources: https://playwright.dev/docs/emulation#devices, https://playwright.dev/docs/emulation#viewport, https://playwright.dev/docs/emulation#ismobile
- Touch-capable context: `touchscreen.tap()` / `page.tap()` require a context initialized with `hasTouch: true`; device descriptors normally provide this, or use `test.use({ hasTouch: true, isMobile: true, viewport: ... })` for custom mobile-landscape Chromium. `isMobile` affects meta viewport behavior and touch events; official docs note it is part of device emulation and not supported in Firefox, so keep this mobile project Chromium/WebKit-oriented rather than assuming Firefox parity. Sources: https://playwright.dev/docs/api/class-touchscreen, https://playwright.dev/docs/api/class-testoptions#test-options-has-touch, https://playwright.dev/docs/api/class-testoptions#test-options-is-mobile
- `touchscreen.tap()` limitation: it dispatches exactly one `touchstart` and `touchend` at one CSS-pixel coordinate and the `Touchscreen` class is “limited to emulating tap gestures.” Therefore it cannot represent “left thumb holds/drags joystick while right thumb taps/interacts” by itself. Use it only for one-finger taps such as accepting a fullscreen prompt button or pressing a standalone interact button; for simultaneous joystick + interact, dispatch touch events manually. Sources: https://playwright.dev/docs/api/class-touchscreen#touchscreen-tap, https://playwright.dev/docs/touch-events
- Manual simultaneous touches pattern: official legacy touch docs recommend `locator.dispatchEvent('touchstart' | 'touchmove' | 'touchend', { touches, changedTouches, targetTouches })` with stable `identifier` values. Adapt the official pinch example into a joystick+interact helper: touch id `0` starts on the left joystick center and moves/holds to the drag vector; touch id `1` starts on the right interact button while id `0` remains in `touches`; end id `1` first if testing tap-while-hold, then end id `0`. Use CSS-pixel viewport coordinates, not Phaser world coordinates, unless converted through the canvas bounding box. Source: https://playwright.dev/docs/touch-events
- Caveat for manually dispatched events: Playwright official docs state `locator.dispatchEvent()` does **not** set `Event.isTrusted`; if the app checks `event.isTrusted`, tests must disable/avoid that check or use real browser input where possible. For Phaser canvas touch handling this is usually acceptable if game logic reads touch coordinates/identifiers rather than `isTrusted`. Source: https://playwright.dev/docs/touch-events
- Reliable GitHub examples for multi-touch construction if official docs are not enough: SVG-Edit creates a `TouchEvent('touchstart')` with two `new Touch({ identifier: 1/2, target, clientX, clientY, screenX, screenY })` entries in `changedTouches`, proving the stable-identifier shape used by browser tests (https://github.com/SVG-Edit/svgedit/blob/master/tests/e2e/unit/svgcore-touch.spec.js#L66); Fabric.js unit tests build `new Touch({ identifier, target, clientX, clientY })` then `new TouchEvent('touchstart', { touches: [touch], changedTouches: [touch] })` (https://github.com/fabricjs/fabric.js/blob/master/src/canvas/Canvas.spec.ts#L63); TaskNotes Playwright e2e tests use `page.evaluate(({ x, y }) => { const element = document.elementFromPoint(x, y); ... new TouchEvent('touchstart'/'touchend', { bubbles: true, cancelable: true, touches, changedTouches }) })` for iPad-style interactions (https://github.com/callumalpass/tasknotes/blob/main/e2e/issues/issue-959-ipad-date-picker-vanishes.spec.ts#L83).
- Practical helper shape for T7 mobile controls (do not implement yet): `const canvas = page.locator('canvas'); await canvas.dispatchEvent('touchstart', { touches: [joyTouch], changedTouches: [joyTouch], targetTouches: [joyTouch] }); await canvas.dispatchEvent('touchmove', { touches: [joyMoved], changedTouches: [joyMoved], targetTouches: [joyMoved] }); await canvas.dispatchEvent('touchstart', { touches: [joyMoved, interactTouch], changedTouches: [interactTouch], targetTouches: [joyMoved, interactTouch] }); await canvas.dispatchEvent('touchend', { touches: [joyMoved], changedTouches: [interactTouch], targetTouches: [joyMoved] }); await canvas.dispatchEvent('touchend', { touches: [], changedTouches: [joyMoved], targetTouches: [] });`. Include `bubbles: true`/`cancelable: true` if using `page.evaluate(new TouchEvent(...))`; with `locator.dispatchEvent`, pass event-init fields directly.
- Fullscreen prompt refusal/acceptance testing: Playwright has no special “accept fullscreen permission prompt” API in these docs. Treat fullscreen as app/browser state, not a native dialog: click/tap the game’s fullscreen affordance from a user gesture, then assert debug fields that distinguish `requested`, `entered`, `failed`, `unsupported`, `left`, or `denied/refused`. To force deterministic refusal/acceptance in T15 hardening tests without production changes, use `page.addInitScript()` before navigation to replace `Element.prototype.requestFullscreen` with a function that rejects or resolves and optionally dispatches expected app-observed events; official Playwright docs support `page.addInitScript()` for pre-load browser mocks, and `page.evaluate()` for reading browser globals. Source: https://playwright.dev/docs/evaluating#init-scripts

Sources: https://playwright.dev/docs/evaluating, https://playwright.dev/docs/test-assertions#expectpoll
